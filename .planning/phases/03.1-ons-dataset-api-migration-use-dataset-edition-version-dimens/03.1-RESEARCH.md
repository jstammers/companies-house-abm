# Phase 03.1 Research: ONS dataset API migration

**Date:** 2026-04-27
**Phase:** 03.1-ons-dataset-api-migration-use-dataset-edition-version-dimens
**Scope:** ONS dataset/edition/version/dimension/observation endpoint migration

## Research Objective

Determine how to migrate `uk_data` ONS retrieval from mixed/legacy retrieval paths to a single dataset API contract centered on:

- `/datasets/{dataset}/editions/{edition}/versions/{version}`
- `/datasets/{dataset}/editions/{edition}/versions/{version}/dimensions/...`
- `/datasets/{dataset}/editions/{edition}/versions/{version}/observations`

while preserving current caller contracts (`UKDataClient` + `ConceptResolver` + adapter semantics from Phase 03).

## Current State (Observed in Code)

1. `ONSAdapter.fetch_series()` currently uses `fetch_sdmx_series(...)` for manifest-backed series, with a fallback stack in `ons.py` that still contains Zebedee-oriented path logic and helper functions (`_fetch_timeseries`, `_SERIES_URI` mappings).
2. `ons_provider.py` delegates to `pandasdmx.Request(...).data(...)` and normalizes the payload; this obscures the dataset/edition/version/observation path contract and couples transport details to pandasdmx behavior.
3. Phase 03 already locked date-window behavior (`start_date`/`end_date`, inclusive bounds, filter-before-limit). Phase 03.1 must preserve this behavior end-to-end.
4. Existing tests are strong on adapter behavior and kwargs forwarding, but do not explicitly enforce direct dataset endpoint composition and retrieval-path migration constraints.

## Recommended Direction

### 1) Establish explicit ONS dataset endpoint contract in manifest/provider

Add/confirm manifest metadata required to build dataset endpoint requests directly (dataset id, edition id, version resolution mode, and any required dimension map), then implement provider logic that fetches observations through dataset endpoints.

### 2) Keep adapter-level behavior contract stable

`ONSAdapter.fetch_series()` should keep the same public behavior:

- same accepted `series_id` values
- same `start_date`/`end_date` and `limit` semantics (window first, then limit)
- same fallback point-series behavior for non-observation handlers (e.g., `HP7A`, `D7RA`)

Only transport path changes; caller semantics remain stable.

### 3) Replace legacy retrieval path usage for manifest-backed observations

Manifest-backed series must no longer rely on legacy retrieval helper paths in `ons.py`. Keep deprecated or unsupported fallback paths only where no dataset observations endpoint supports the series.

### 4) Keep no-new-dependency posture

Prefer stdlib/`uk_data._http` transport patterns already used across adapters. Do not introduce additional runtime libraries for the migration.

## Testing Strategy

1. **Provider unit tests (`test_ons_provider.py`)**
   - verifies endpoint param construction includes dataset/edition/version/observation request shape
   - verifies `start_date`/`end_date` propagate to request filters
   - verifies payload normalization into `{date, value}` rows

2. **Adapter unit tests (`test_ons.py`)**
   - verifies `ONSAdapter.fetch_series()` still returns canonical `TimeSeries`
   - verifies inclusive windowing and filter-before-limit remain true after transport change
   - verifies fallback handlers still return point-series behavior

3. **Integration tests (`test_ons_integration.py`)**
   - keep network-gated integration checks for all advertised series
   - ensure adapter can fetch all `available_series()` IDs with live endpoint routing

## Risks and Mitigations

1. **Risk:** Endpoint or version variability across datasets.
   **Mitigation:** Resolve version through explicit dataset version endpoint and keep deterministic default/fallback behavior in provider.

2. **Risk:** Behavior regression in limit/date-window precedence.
   **Mitigation:** Preserve Phase 03 semantics with explicit regression tests (`window -> limit`).

3. **Risk:** Hidden dependency on legacy helper paths.
   **Mitigation:** Remove/retire legacy retrieval calls from manifest-backed series code path and add tests asserting provider call path.

## Architectural Responsibility Map (for planning)

- **Manifest layer (`ons_manifest.py`)**: endpoint metadata contract
- **Provider layer (`ons_provider.py`)**: HTTP retrieval + normalization from dataset observation payloads
- **Adapter layer (`ons.py`)**: canonical `TimeSeries` mapping, date-window filtering contract, fallback behavior
- **Tests (`tests/adapters/*`)**: migration regression guardrails

## Out of Scope

- New data providers
- UI changes
- ABM model behavior changes
- Cross-source query abstractions
